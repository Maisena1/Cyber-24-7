Introducción

Contexto del trabajo

Este trabajo se entrega como parte de la materia Práctica Profesionalizante II, en el segundo año de la carrera “Tecnicatura en Análisis y Desarrollo de Software” del Instituto de Profesorado Sedes Sapientiae. El proyecto utiliza el método de diseño orientado a objetos para solventar un problema o necesidad.

Objetivos generales

Permite substanciales ahorros de tiempo al proveer los servicios del local. Automatizando procesos y diferentes categorías de equipos para diferentes usos (streaming, gaming, simuladores, etc), cada uno con un arancel específico.

Proceso de resolución

  • 1. Análisis de la Situación: Evaluación de la gestión manual de las PCs en cybers, identificación de problemas como errores, desorden y pérdida de tiempo. Se hicieron entrevistas para hacer el análisis.
  • 2. Definición de Requisitos Funcionales, funcionalidades básicas y claves:Dar de alta un equipo; Manejar reservas múltiples; Listar próximos mantenimientos.
  • 3. Creación de Diagramas UML: Visualización del sistema mediante diagramas UML para facilitar el entendimiento y la planificación del sistema.
  • 4. Planificación del Proyecto: Establecimiento de un enfoque metódico y ordenado, siguiendo prácticas de empresas de software reales
  • 5.Programación del Sistema: Implementación del software respetando las definiciones y requisitos previamente establecidos.
  • 6. Visión a Futuro: Consideración de la posibilidad de incorporar nuevas funciones en el futuro (por ejemplo, un sistema de promociones para clientes frecuentes).
  • 7. Trabajo en Equipo: Colaboración entre los miembros del grupo, aprovechando las habilidades individuales para contribuir al proyecto de manera efectiva.

Contexto del problema

La gestión tan diversa que tienen los cibers a pesar de su similitud y el desorden que esto generaba fue el detonante de la elección de la temática para el proyecto de esta materia.

Marco Teórico

1. Orientación a Objetos

Como explican Kendall y Kendall en su libro (Análisis y Diseño de Sistemas), la orientación a objetos es una forma de pensar el software usandoobjetos que representan cosas del mundo real, cada uno con sus propios datos (atributos) y acciones (métodos)​. La gran ventaja de este enfoque es que todo queda organizado de manera lógica, y además se puede reutilizar código o ampliarlo sin romper el sistema completo. En nuestro caso, aplicamos este modelo creando clases comoEquipo , Reserva, Simulador y ComputadoraDeEscritorio. Por ejemplo, cada máquina del ciber tendrá atributos como el número de serie, el estado actual (ocupada, apagada, en mantenimiento) y métodos como ocupar(), apagar(), reservar() o liberar()​. Supongamos que en el futuro, el dueño del ciber quiere agregar consolas de videojuegos para alquiler. Gracias a que todo está basado en orientación a objetos, simplemente podríamos crear una nueva clase ConsolaDeVideojuego que herede de Equipo, sin necesidad de reprogramar el sistema entero. Esto muestra por qué elegimos este paradigma: nos da flexibilidad, orden y nos ahorra mucho tiempo de programación a largo plazo

2. Proyecto de Software

Según Pressman en su libro (Ingeniería de Software), un proyecto de software no es solo programar, sino un esfuerzo organizado que tiene que cumplir ciertos pasos para que el producto final funcione y cumpla con lo que necesita el cliente​. En Cyber24/7 planteamos el proyecto como corresponde: primero hicimos un análisis de la situación en los cibers, donde vimos que la gestión manual de las PCs causaba errores, desorden, y pérdida de tiempo. A partir de ahí, definimos requisitos funcionales como "dar de alta un equipo", "manejar reservas múltiples" o "listar próximos mantenimientos"​. Después hicimos los diagramas UML para visualizar el sistema, y a partir de ahí vamos a programarlo respetando esas definiciones. Cada etapa del proyecto tiene su sentido: no es simplemente "ponerse a programar", sino trabajar con método y planificación, como lo hacen las empresas de software reales. Incluso pensamos el proyecto con visión a futuro: si el ciber quiere incorporar nuevas funciones (por ejemplo, un sistema de promociones para clientes frecuentes), nuestra organización como proyecto de software nos permitirá agregarlo de manera ordenada.

3. UML (Unified Modeling Language)

UML, según Kendall y Kendall, es un lenguaje de modelado gráfico que nos permite ver cómo será el sistema antes de construirlo​. No solo sirve para planificar mejor, sino también para que cualquier persona que vea los diagramas entienda cómo funciona el sistema, aunque todavía no haya una sola línea de código escrita. En Cyber24/7 usamos UML para diseñar el sistema desde varios ángulos:

  • Creamos diagramas de casos de uso que muestran qué acciones puede hacer un usuario o un empleado del ciber, como "extender el horario de uso" o "cancelar una reserva".
  • Armamos un diagrama de clases para definir qué entidades existen (Equipo, Reserva, Sesión) y qué relaciones hay entre ellas​.
  • También hicimos diagramas de estados para representar cómo cambia la situación de una computadora a lo largo de su vida útil (por ejemplo, pasar de "ocupada" a "bloqueada" si se acaba el tiempo de sesión).
Estos diagramas nos sirven de guía en todo el proyecto y nos aseguran que el equipo de desarrollo (o cualquier programador nuevo que se sume) pueda entender cómo está armado el sistema.

4. Proyecto bajo Metodologías Ágiles

Hoy en día, desarrollar software es muy diferente a como se hacía hace 20 años. Ya no se define todo de entrada, sino que se trabaja de manera ágil, adaptándose a los cambios. El documento sobre metodologías ágiles que leímos explica que estas metodologías se centran en el cliente, la flexibilidad y el trabajo en equipo, haciendo entregas parciales y mejorando a partir de la retroalimentación​. En Cyber24/7 aplicamos esta lógica. Vamos a trabajar en sprints cortos donde entregamos versiones funcionales del sistema. Por ejemplo, en la primera entrega nos enfocaremos en que el sistema permita registrar computadoras y gestionar sesiones básicas. Después, gracias al feedback, podemos ir agregando mejoras como la reserva por tipo de computadora (gama baja, media o alta). También, si durante el desarrollo el cliente nos dice que necesita un reporte de uso por horarios, podremos adaptar el proyecto sobre la marcha, sin tirar abajo todo lo que ya hicimos. Trabajar con metodologías ágiles nos permite:

  • Adaptarnos a necesidades reales y no suposiciones.
  • Entregar valor rápido en cada sprint.
  • Corregir errores tempranamente.
  • Mantener la motivación del equipo, viendo resultados frecuentes.

Administración de equipos

  • Cambiar el estado de un equipo (alta, baja temporal, baja permanente)
    • Entradas: Equipo.numero_de_serie, Equipo.estado
    • Salidas: Equipo.estado
    • Descripción: El personal desea controlar los estados de las computadoras. Se debe especificar si se quiere dar de baja temporal (por mantenimiento), baja permanente (por pérdida total) o de alta (en funcionamiento).
    • Alternativas:
      • Transición de estado inválida (eg: se trató de dar de alta un equipo en baja permanente), se le da una justificación al personal.
      • Equipo inexistente. Se le comunica al personal.
    • Buscar equipos registrados (listando todos, filtrando según su estado de equipo)
      • Entradas: Equipo.numero_de_serie, Equipo.estado
      • Salidas: -
      • Descripción: El personal desea ver los equipos que están de alta, baja temporal o permanente. Luego de seleccionar un filtro adecuado (o no), se puede elegir una computadora en particular para acciones futuras
      • Alternativas:
        • No hay equipos disponibles. Se le muestra eso al personal.
    • Listar próximos mantenimientos
      • Entradas: Equipo.numero_de_serie, Equipo.fecha_de_ultimo_mantenimiento, Equipo.estado
      • Salidas: Equipo.estado, Equipo.fecha_de_ultimo_mantenimiento
      • Descripción: El personal quiere registrar los mantenimientos, debe indicar en qué fecha serán realizados para poder darla de baja de manera temporal y cambiar la fecha de último mantenimiento.
      • Alternativas:
        • No hay próximos mantenimientos. Se le informa al personal.
        • No hay equipos a mantener. Se le muestra lo mismo al personal.
    • Agregar equipo (darlo de alta)
      • Entradas: Equipo.numero_de_serie
      • Salidas: Equipo.numero_de_serie, Equipo.anno_de_compra, Equipo.fecha_de_ultimo_mantenimiento, Equipo.estado, Equipo.situacion, Equipo<-Calidad, uno de Simulador.{tipo_de_simulacion, anno, empresa, modelo} o ComputadorDeEscritorio.{procesador, almacenamiento, placa_de_video, ram}
      • Descripción: El personal quiere agregar un equipo. Debe dar todas sus características, además de especificar si es un simulador o una PC.
      • Alternativas:
        • El equipo ya está registrado. Se le informa al personal.
        • Error de formato en el formulario. Se le informa al personal, indicando el campo inválido.

Gestión de sesiones

  • Extender horario permitido
    • Entradas: Sesion.id_de_sesion, Reserva.inicio
    • Salidas: Sesion.hora_de_bloqueo
    • Descripción: Al cliente se le bloquea su computadora debido a que se le acabó el tiempo de uso asignado, el personal puede extender su horario si el cliente así lo desea.
    • Alternativas:
      • No existe la sesión. Se notifica al personal.
      • El horario permitido se solapa con una reserva. No se permite extender el horario en esa máquina.
    • Cambiar la situación del equipo (ocupado, desocupado, bloqueado, apagado)
      • Entradas: Equipo.numero_de_serie, Equipo.situacion
      • Salidas: Equipo.situacion
      • Descripción: El personal desea cambiar la situación de una computadora, generalmente para marcar una máquina como ocupada o desocupada. Puede hacerlo indicando la máquina específica y la situación a la que se desea pasar.
      • Alternativas:
        • Transición de estado inválida (eg: se trató de ocupar un equipo apagado)
      • Buscar equipo (listando todos, filtrando según su situación)
        • Entradas: Equipo.numero_de_serie, Equipo.situacion
        • Salidas: -
        • Descripción: El personal desea ver las computadoras basadas en un filtro de estado de sesión. Debe indicar el filtro adecuado y seleccionar una computadora de ser necesario.
        • Alternativas:
          • No hay equipos. Se le avisa al personal.

Manejo de reservas

  • Reservar un conjunto de equipos (especificando cantidad de máquinas por tipo/gama/simulador y fecha de inicio/finalización)
    • Entradas: Reserva.inicio, Reserva.fin
    • Salidas: Reserva.inicio, Reserva.fin, Reserva--Equipo (1 o más)
    • Descripción: Un futuro cliente desea realizar un evento o reservar varias computadoras, informando al personal la cantidad de computadoras y en qué fecha se requerirán para que estos registren la reserva. Nótese que el personal debe especificar cantidades y no máquinas específicas, el sistema las asigna.
    • Alternativas:
      • El personal cometió errores al completar partes del formulario. Se les informa dónde y se permite su corrección.
      • No hay equipos disponibles para el periodo elegido. Se le informa al personal.
  • Listar reservas
    • Entradas: Reserva.id_de_reserva, Reserva.inicio, Reserva.fin
    • Salidas: -
    • Descripción: El personal desea ver las reservas registradas. Se muestran todas las reservas futuras de todas las computadoras.
    • Alternativas:
      • No hay reservas. Se le avisa al personal.
  • Cancelar reserva
    • Entradas: Reserva.id_de_reserva
    • Salidas: -
    • Descripción: El cliente le informa al personal que quiere cancelar una reserva. El personal la cancela desde el sistema.
    • Alternativas:
      • No existe tal reserva. Se le avisa al personal.
      • El momento reservado ya pasó. No se cancela la reserva.
